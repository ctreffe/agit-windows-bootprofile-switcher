# v1.0.0 Release Scope

This document defines the intended scope for the initial stable BootProfile
Switcher release.

## Release Intent

`v1.0.0` is an initial stable foundation release. It proves that BootProfile
Switcher can identify a selected managed Windows boot profile, validate a
machine-wide profile configuration, dispatch configured modules before user
logon and log both action and no-op paths predictably.

It is not intended to be the first production release with real operational
system-changing modules.

## Included

The release scope includes:

- reversible managed Windows Boot Manager entries for Mode A and Mode B
- GUID-based current boot profile resolution with description fallback
- normal unmanaged Windows startup as a successful no-op case
- startup Scheduled Task integration
- machine-wide `profiles.json` configuration under ProgramData
- configuration validation and known-bad fixtures
- configuration-driven profile engine module dispatch
- explicit no-op behavior for missing, invalid or incomplete configuration
- structured startup logging and module action logging
- the harmless `validation-log` module
- the temporary `demo-system-marker` module for release demonstration
- combined demo setup wrappers for the current v1.0.0 flow
- documented install, validation and uninstall workflows

## Excluded

The release scope explicitly excludes:

- production modules that change networking, services, policies, firewall rules
  or other Windows behavior
- custom PowerShell script execution from profile configuration
- variable managed profile counts beyond the current Mode A and Mode B proof
  shape
- hiding or renaming the default Windows boot entry
- Group Policy packaging or ADMX/ADML policy templates
- a graphical user interface

## Temporary Demo Module

`demo-system-marker` is included only to demonstrate that configured modules can
perform a real, harmless system-level write.

It writes:

```text
C:\ProgramData\BootProfileSwitcher\runtime\demo-current-profile.json
```

The marker records the resolved mode, name, identifier and timestamp. It does
not change Windows behavior. The marker file can be deleted safely.

The module should be removed or replaced after `v1.0.0` once production modules
exist.

## Safety Expectations

The release must preserve these safety properties:

- boot menu installation is reversible through `uninstall.cmd`
- startup hook installation is reversible through `uninstall-startup-hook.cmd`
- the combined demo setup is reversible through `uninstall-demo.cmd`
- configuration installation validates before writing to ProgramData
- missing or invalid configuration must not execute modules
- unmanaged Windows startup must not execute modules
- module execution must be visible in `logs/module-actions.log`
- startup decisions must be visible in `logs/startup-profile.log`

## Release Readiness

Before tagging `v1.0.0`, confirm:

- the v0.9 validation checklist remains passing for the current runtime path
- `demo-system-marker` has been validated through at least one managed boot
  profile runtime path
- README and README.de describe setup, validation, logs, cleanup and current
  limitations
- combined demo setup wrappers preserve the granular setup and teardown scripts
- PROJECT_CONTEXT and CHANGELOG reflect the release scope and current status
